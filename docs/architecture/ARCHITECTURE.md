# Architecture

>
> This document describes the architectural direction planned for the next major version of the project.
>
> It is not intended to be a complete specification. Instead, it records the principles, responsibilities and design decisions that will guide the incremental modernization of the application.

---

# Purpose

Version **2.0** aims to modernize the technical foundation of the ETL pipeline while preserving the business rules that have already been validated through long-term production use.

The objective is **not** to rewrite the application.

Instead, the goal is to progressively replace technical components that currently introduce unnecessary coupling, making the application easier to evolve, maintain and extend.

---

# Current Architecture

The current implementation is centered around the HTML structure generated by the source system.

Its execution flow can be summarized as follows:

```
Legacy System
      │
      ▼
HTML
      │
      ▼
BeautifulSoup
      │
      ▼
Known Fields
      │
      ▼
Microsoft Access
```

This architecture has successfully supported the application throughout its operational life.

However, several limitations have become evident over time:

- extraction depends on previously identified HTML elements;
- new fields require manual implementation;
- changes in the source system can easily break extraction;
- persistence is strongly coupled to the extracted schema.

These limitations motivated the architectural evolution described in this document.

---

# Architectural Vision

Version **2.0** introduces a fundamental shift in the application's perspective.

Instead of depending on the HTML representation of a document, the application will operate on the document itself.

The acquisition technology becomes an implementation detail.

Regardless of whether information originates from HTML, PDF or another format introduced in the future, the remainder of the application should operate identically.

The intended processing flow becomes:

```
Document
     │
     ▼
Extraction
     │
     ▼
JSON
     │
     ▼
Persistence
```

Business logic should never depend on how the document was obtained.

---

# Architectural Principles

## Documents over technologies

The application should not depend on HTML, BeautifulSoup or any specific extraction technology.

These components become implementation details rather than architectural foundations.

Future extraction mechanisms should integrate naturally into the same processing flow without affecting business logic.

---

## Field-oriented extraction

Instead of searching for predefined HTML identifiers, extraction should identify semantic key/value relationships contained within the document.

Example:

```
Name
John Smith
```

Result:

```json
{
    "name": "John Smith"
}
```

This approach allows previously unknown fields to be captured without requiring source code modifications.

Extraction is based on the visual organization of the document instead of technology-specific markup such as HTML elements or identifiers.

---

## Dynamic document structure

The application should no longer assume that every document contains the same fields.

Each processed document may expose different information.

Consequently, persistence should evolve to support variable document structures instead of relying on a fixed relational schema.

---

## Incremental evolution

Architectural modernization must remain incremental.

Each branch should produce a coherent, reviewable and technically stable increment. Whenever possible, the application should remain functional throughout the modernization process. Transitional branches that establish architectural foundations without immediately restoring end-to-end functionality are acceptable, provided they preserve a clear migration path and avoid disruptive rewrites.

No stage of the project should require a complete rewrite before becoming usable.

---

# Core Components

The future architecture is intentionally based on a small number of responsibilities.

The goal is to improve cohesion without introducing unnecessary abstraction.

---

## Target (`Alvo`)

Represents a pending item to be collected.

Its responsibility is limited to locating and obtaining the corresponding document.

Typical responsibilities include:

- identifying pending records;
- requesting document generation;
- retrieving the original HTML.

---

## Page (`Pagina`)

Represents a single document being processed by the application.

This object becomes the central element of the extraction pipeline.

A page may contain:

- original HTML;
- rendered PDF;
- extracted data;
- processing metadata.

Typical operations include:

- rendering;
- semantic extraction;
- metadata generation.

Rather than creating multiple intermediate objects (Document, Line, Word, etc.), the application intentionally concentrates document-related responsibilities within this component to preserve implementation simplicity.

---

## Batch (`Lote`)

Represents a collection of processed pages.

Its primary responsibilities are:

- grouping extracted documents;
- reducing persistence operations;
- organizing processing flow.

Batching remains an implementation detail intended to improve efficiency rather than part of the business model.

---

## Persistence (`Persistencia` / `Banco`)

Responsible exclusively for storing processed information.

Persistence should remain independent of document acquisition and extraction.

Future database technologies should be replaceable without affecting the remainder of the pipeline.

---

# Expected Execution Flow

The expected execution sequence becomes:

```
Target List

      │

      ▼

Target.fetch_page()

      │

      ▼

Page

      │

render()

      │

extract()

      │

      ▼

JSON

      │

      ▼

Batch

      │

persist()

      │

      ▼

Database
```

Each stage should perform a single responsibility while remaining independent of the implementation details of neighboring stages.

---

# Configuration

Configuration should no longer describe document structure.

Information such as:

- HTML element identifiers;
- element types;
- extraction rules specific to HTML;

should disappear from the configuration layer.

Configuration should instead describe only execution parameters, including:

- directories;
- database selection;
- processing limits;
- user interface behavior;
- logging options;
- execution settings.

This allows extraction strategies to evolve independently of application configuration.

---

# Version 2.0 Goals

The primary goals of this architectural evolution are:

- remove the dependency between extraction and HTML structure;
- introduce semantic document extraction;
- support documents with variable structures;
- simplify future persistence changes;
- reduce architectural coupling;
- improve long-term maintainability.

---

# Out of Scope

Version **2.0** is **not** intended to:

- transform the application into a generic framework;
- introduce dozens of small classes solely to follow architectural patterns;
- adopt design patterns without practical justification;
- increase abstraction at the expense of readability;
- sacrifice simplicity in pursuit of theoretical purity.

Architectural decisions should always prioritize maintainability over architectural sophistication.

---

# Guiding Philosophy

The application exists to solve a practical operational problem.

Every architectural decision should therefore satisfy one or more of the following objectives:

- reduce coupling;
- simplify maintenance;
- improve robustness;
- facilitate future evolution;
- preserve validated business behavior.

Whenever these objectives conflict with architectural complexity, simplicity should be preferred.

---

This document represents the current architectural direction of the project.

Implementation details are expected to evolve throughout development.

However, future changes should preserve the principles established here, ensuring that the modernization effort remains coherent across all Version **2.x** iterations.
